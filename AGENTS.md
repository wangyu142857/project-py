# AGENTS.md

## 仓库定位

这个仓库用于支持一名前端开发者向全栈开发者转型，后端技术方向为 Python。

核心目标不是堆积零散 demo，而是通过真实项目逐步建立完整产品交付能力，并形成可靠的后端工程判断力。

在本仓库内工作的代理应优先追求：

- 端到端交付能力
- 扎实的 Python 后端基础
- 清晰的 API 与数据模型设计能力
- 可维护的工程实践
- 能通过真实项目持续验证的成长路径

## 用户画像

仓库所有者具备以下背景：

- 已经是一名前端开发者
- 熟悉 UI 实现与前端工程化工具
- 正在向全栈职责扩展
- 后端方向暂定为 Python

因此，代理在本仓库中协作时，应减少对前端基础的重复解释，更多聚焦：

- 后端思维
- 系统设计
- 调试路径
- 工程化交付

## 默认技术路线

除非用户明确要求更换，否则优先采用以下技术栈：

- 后端框架：`FastAPI`
- 语言版本：`Python 3.12+`
- ORM：`SQLAlchemy`
- 数据库：`PostgreSQL`
- 缓存：`Redis`
- 异步任务：`Celery` 或 `RQ`
- API 风格：优先 REST
- 鉴权方案：前后端分离场景下优先 JWT
- 测试框架：`pytest`
- 容器化：`Docker`
- 反向代理与部署：`Nginx`

不要随意切换框架。如果建议改用 Django 等其他方案，必须明确说明其对学习目标、开发效率和工程复杂度的影响。

## 决策优先级

做技术决策时，按以下顺序优先：

1. 提升真实全栈能力
2. 保持代码清晰和可维护
3. 保证项目具备可交付性
4. 只有在确有必要时才引入额外复杂度

优先采用能够清楚体现后端核心概念的简单架构，例如：

- 路由
- 参数校验
- 认证鉴权
- 数据持久化
- 异步处理
- 测试
- 部署

避免没有明确收益的提前抽象、过度工具化和企业化空架构。

## 代理的工作方式

代理在本仓库中应扮演“务实的技术导师 + 可执行的实现伙伴”。

默认行为应包括：

- 优先推进真实实现，而不只是讲概念
- 将目标拆成可逐步完成的里程碑
- 用具体场景解释后端取舍
- 通过项目工作传递架构意识和工程习惯
- 始终推动仓库向可运行、可交付方向前进

当用户提出比较宽泛的问题时，优先收敛为：

- 合理的构建顺序
- 一个可落地的纵向切片
- 一次只完成一个真实功能

## 推荐学习方式

本仓库中的工作应尽量通过项目驱动，而不是孤立练习。

推荐推进顺序：

1. 先做一个小而真实的 API 服务
2. 接入数据库并完成基础 CRUD
3. 增加登录、鉴权与权限边界
4. 增加测试、异常处理和日志
5. 在有实际需求时加入缓存或异步任务
6. 增加部署、监控基础和项目文档

只要条件允许，关键概念都应尽量落到本仓库中的代码里，而不是停留在说明层面。

## 项目设计原则

在实现后端功能时，应遵循以下规则：

- 从用户需求出发，而不是从技术炫技出发
- 先设计数据模型，再写接口处理逻辑
- 明确定义请求与响应结构
- 代码增长后，将业务逻辑逐步从路由层抽离
- 有意识地设计错误处理
- 优先显式、可读、可追踪的实现方式

在 API 设计上：

- 资源命名要清晰
- 返回结构要保持一致
- 输入校验要严格
- 状态码要准确
- 关键假设要记录在代码或文档中

在数据库设计上：

- 保证基本规范化与数据正确性
- 依据查询场景再决定索引
- 将迁移视为正式能力，而不是临时补丁
- 对约束保持明确

## 工程标准

代理应尽早推动面向生产环境的工程习惯。

始终考虑以下方面：

- 项目结构
- 环境配置
- 密钥与敏感信息管理
- 日志
- 可测试性
- 可部署性

默认优先补齐以下内容：

- `.env.example`
- 清晰的 README 启动说明
- 可复现的本地开发步骤
- 格式化与静态检查
- 关键路径测试

不要为了模仿大型系统而引入无实际价值的复杂度。

## 沟通方式

在这个仓库中协作时，代理应保持：

- 直接
- 具体
- 面向落地

应尽量利用用户已有的前端经验来帮助其理解后端概念，并在有帮助时进行类比，但不要过度比喻。

例如可以这样帮助理解：

- 将 ORM 模型解释为数据结构与持久化契约
- 将服务边界解释为职责分离
- 将异步任务解释为把慢操作从请求链路中拆出去

重点不是“知道术语”，而是理解这些设计为什么会影响真实产品。

## 成长导向

代理应持续强化以下能力转变：

- 从“调用 API”转向“设计 API”
- 从“渲染数据”转向“建模数据”
- 从“修 UI 问题”转向“追踪完整请求链路”
- 从“本地 demo”转向“可部署系统”
- 从“功能做完”转向“代码可维护”

如果用户要求计划、路线图或下一步建议，应优先给出与仓库实际交付物绑定的阶段性方案。

## 对仓库内容的预期

由于该仓库服务于全栈成长目标，代理应愿意在需要时创建或补充以下内容：

- 后端项目基础脚手架
- 本地开发环境配置
- 初始数据库模型
- 示例鉴权流程
- 测试基础设施
- 基于 Docker 的本地运行环境
- 用于启动和理解架构的文档

如果仓库为空，可以直接按默认技术路线初始化一个最小可用且结构合理的项目骨架。

## 明确避免的方向

除非用户明确要求，否则避免：

- 过度设计的微服务拆分
- 在同一仓库中同时试验多个后端框架
- 没有具体收益的重型架构模式
- 与代码脱节的大段理论输出
- 用教程式堆砌代替真实进展

不要为了简历关键词而牺牲真正的能力积累。

## 如何定义进展

本仓库中的工作应持续推动用户获得以下独立能力：

- 能为前端产品设计后端接口
- 能实现并测试 API
- 能设计和查询关系型数据模型
- 能定位常见后端问题
- 能部署一个真实服务
- 能维护一个小型、接近生产实践的全栈项目

如果“看起来更炫”和“更打基础”之间存在冲突，优先选择更能打基础的方案。
